## 3. Patient portal expansion

New tables (all under RLS; patients read their own rows, staff read all, doctors write clinical rows):

- `profiles` — one row per user: full name, DOB, gender, blood group, medical aid name + number, emergency contact name + phone, allergies text, chronic conditions text. Auto-created on signup via trigger; patients edit their own.
- `prescriptions` — appointment_id, patient_id, doctor_id, medication, dosage, instructions, issued_at.
- `lab_results` — patient_id, appointment_id (nullable), title, result_summary, file_url (nullable), performed_at.
- `visit_notes` — appointment_id, patient_id, doctor_id, symptoms, diagnosis, treatment, follow_up_date, notes. One per appointment.
- `invoices` — appointment_id, patient_id, amount, payment_method, payment_status (`unpaid|paid|refunded`), issued_at, paid_at. Auto-created on appointment insert (unpaid). Marked paid when appointment is confirmed by staff.

Portal routes (all under `_authenticated/portal.*`):

- `portal.tsx` — dashboard shell + sidebar nav (Profile, Appointments, Medical History, Prescriptions, Lab Results, Invoices).
- `portal.profile.tsx` — view + edit profile.
- `portal.appointments.tsx` — existing page, add "Download confirmation" (renders + prints the confirmation), "Rebook" (prefills booking form).
- `portal.history.tsx` — read-only medical history: past visits with diagnosis/treatment/prescriptions, allergies, chronic conditions.
- `portal.prescriptions.tsx` — list.
- `portal.lab-results.tsx` — list with download links.
- `portal.invoices.tsx` — list with status, print-friendly invoice view including bank details for unpaid EFT invoices.

Header gets a "My Portal" dropdown when signed in.

## 4. Doctor portal (clinical workflow)

Routes under `_authenticated/staff.*`, gated by `is_staff`:

- `staff.appointments.tsx` — existing status controls + a new "Open patient file" action per row.
- `staff.patients.tsx` — searchable patient list (from `profiles` + appointments).
- `staff.patient.$id.tsx` — patient file: profile, appointment history, prescriptions, lab results, past visit notes.
- `staff.visit.$appointmentId.tsx` — current-visit form: symptoms, diagnosis, treatment, follow-up date, notes; add prescription rows; add lab result rows. Saving marks the appointment `completed`.

Doctor notification: on booking insert, a `staff_notifications` row is created (new_appointment) — displayed as a badge + list on the staff dashboard. (Email to doctor is out of scope this phase.)

## Security notes

- Bank account is only exposed to the authenticated patient who booked (and to staff). Public site never shows it. A visible banner on the confirmation screen restates that this is a school-project account and warns against real-world use.
- All new tables get `GRANT`s to `authenticated` + `service_role`, RLS enabled, and policies scoped via `auth.uid()` / `is_staff` / `has_role('doctor')`.
- Clinical write policies (visit_notes, prescriptions, lab_results) require `has_role('doctor')`. Patients get read-only on their own rows.
- Profile updates restricted to `auth.uid() = user_id`.
